# Research: Dedup Permission Layer (feature 008 №20)

Unknowns из Technical Context отсутствуют — правка внутренняя, без новых зависимостей и без изменения схемы. Ниже зафиксированы принятые решения по двум узлам.

## Decision 1 — «Выживший» среди дублей проверки прямого доступа

**Decision**: Оставить `can_write`, удалить `has_direct_access` и убрать её из импорта `database/db.py`.

**Rationale**: Поиск по кодовой базе (`grep can_write|has_direct_access`) показал:
- `can_write` — live: `services/ui_service.py:330`, `services/permission_service.py:96`, `tests/test_journeys/*` (3 файла).
- `has_direct_access` — 0 live-вызовов: определение `database/permissions.py:74` + единственный импорт `database/db.py:27`, нигде не вызывается.
Реализации идентичны (`SELECT 1 FROM direct_topic_access WHERE user_id=? AND topic_id=? LIMIT 1`). Удаление неиспользуемой функции — нулевой риск для путей исполнения.

**Alternatives considered**:
- Слить `can_write` → делегирование в `has_direct_access`: отвергнуто — оставляет две функции, дедуп не достигнут, добавляет вызов без пользы.
- Удалить `can_write`, оставить `has_direct_access`: отвергнуто — сломало бы 3 файла живых вызовов.

## Decision 2 — Честная семантика `is_superadmin`

**Decision**: Свести тело к авторитетной проверке `user_id == ADMIN_ID`. Диагностический warning о рассогласовании роли в БД сохранить как опциональную наблюдаемость, вычисляемую ТОЛЬКО когда ID совпал, и НЕ влияющую на возвращаемый результат.

**Rationale**: Текущий код (`services/permission_service.py:12-26`): при `user_id != ADMIN_ID` → `False`; иначе цикл по ролям БД возвращает `True` при наличии роли, а при отсутствии (стр.26) — тоже `True`. Итог: для `ADMIN_ID` результат всегда `True` независимо от БД → БД-ветвь мёртвая по результату, docstring вводит в заблуждение. Честный поток: `ADMIN_ID` — источник истины; результат `user_id == ADMIN_ID`.

**Alternatives considered**:
- Сделать БД-проверку «живой» (требовать роль в БД, иначе `False`): отвергнуто — это **изменение поведения** (ADMIN_ID без роли перестал бы быть суперадмином), выходит за рамки cleanup-спеки и меняет модель безопасности. Спека FR-004 требует сохранить наблюдаемый результат.
- Полностью удалить warning: допустимо (FR-005 — MAY), но сохранение диагностики о ручном рассинхроне ролей полезно; оставляем, отвязав от результата.

**Формулировка docstring**: обновить, чтобы отражала фактическую семантику (ADMIN_ID — источник истины; лог — только диагностика), устранив ложное обещание «дополнительной проверки».
