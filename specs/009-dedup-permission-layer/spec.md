# Feature Specification: Dedup Permission Layer (feature 008 №20)

**Feature Branch**: `009-dedup-permission-layer`

**Created**: 2026-07-13

**Status**: Draft

**Input**: User description: "Feature 008 №20 — устранение дублей/мёртвого кода в слое прав доступа (correctness cleanup, минимальный footprint)."

## User Scenarios & Testing *(mandatory)*

> Это техническая задача по устранению долга в слое прав доступа. «Пользователь» здесь — сопровождающий разработчик/аудитор безопасности; ценность — снижение риска рассинхронизации прав и ложных выводов при аудите. Наблюдаемое поведение для конечных пользователей бота НЕ меняется.

### User Story 1 - Единственная точка проверки прямого доступа (Priority: P1)

Сопровождающий, читая слой прав, должен видеть ровно одну функцию, отвечающую на вопрос «есть ли у пользователя прямой доступ к топику», а не две функции с идентичной реализацией, одна из которых нигде не вызывается.

**Why this priority**: Дублирующая мёртвая функция — источник дрейфа: будущая правка может уйти в неиспользуемую копию, оставив реальную проверку прав нетронутой (риск безопасности) либо наоборот. Устранение дубля убирает эту ловушку.

**Independent Test**: Проверяется удалением мёртвой функции и прогоном полного тест-сьюта — все существующие проверки прав (`test_journeys`) остаются зелёными, поведение живой функции неизменно.

**Acceptance Scenarios**:

1. **Given** пользователь с прямым доступом к топику, **When** система проверяет право записи, **Then** доступ разрешён (как и прежде).
2. **Given** пользователь без прямого доступа, **When** система проверяет право записи, **Then** доступ запрещён (как и прежде).
3. **Given** кодовая база после правки, **When** сопровождающий ищет функции проверки прямого доступа, **Then** находит ровно одну живую функцию; неиспользуемого дубля и его импорта нет.

---

### User Story 2 - Честная семантика проверки суперадмина (Priority: P1)

Аудитор, читающий проверку суперадмина, должен видеть поток управления, соответствующий фактическому поведению: результат определяется авторитетным источником истины, а не мёртвой ветвью, обещающей проверку, которая ни на что не влияет.

**Why this priority**: Текущая функция документирует «дополнительную проверку роли в БД», но возвращает один и тот же результат независимо от наличия роли — читатель делает ложный вывод о защите от ручного вмешательства. Это подрывает доверие к аудиту прав.

**Independent Test**: Характеризационный тест фиксирует наблюдаемый результат (True для авторитетного администратора, False для остальных) до и после правки — он неизменен; при этом код больше не содержит ветви, чей результат не зависит от её условия.

**Acceptance Scenarios**:

1. **Given** авторитетный администратор (совпадение с настроенным идентификатором), **When** проверяется статус суперадмина, **Then** результат положительный (как и прежде), независимо от наличия соответствующей роли в БД.
2. **Given** любой иной пользователь, **When** проверяется статус суперадмина, **Then** результат отрицательный (как и прежде).
3. **Given** авторитетный администратор без соответствующей роли в БД, **When** проверяется статус, **Then** результат положительный и (опционально) фиксируется диагностическое предупреждение о рассогласовании — но предупреждение не меняет результат.

### Edge Cases

- Что происходит, если у авторитетного администратора роль в БД присутствует? → Результат положительный; поведение идентично случаю отсутствия роли (в этом и суть честной семантики).
- Что происходит с внешними вызывающими сторонами удалённой функции? → Их нет (функция мёртвая), поэтому удаление не затрагивает ни одного пути исполнения; тесты подтверждают.
- Меняется ли контракт живой функции проверки прямого доступа? → Нет: сигнатура, возвращаемый тип и результат для всех входов сохраняются.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Слой прав MUST предоставлять ровно одну функцию проверки наличия прямого доступа пользователя к топику; дублирующая функция с идентичной реализацией MUST быть удалена вместе с её импортом в агрегирующем модуле БД.
- **FR-002**: Удаление дублирующей функции MUST NOT изменить наблюдаемый результат проверки прав записи ни для одного входа (пользователь с доступом → разрешено; без доступа → запрещено).
- **FR-003**: Проверка статуса суперадмина MUST иметь поток управления, в котором отсутствует ветвь, чей результат не зависит от её собственного условия (мёртвая ветвь устранена).
- **FR-004**: Наблюдаемый результат проверки суперадмина MUST остаться прежним: положительный для авторитетного администратора, отрицательный для остальных.
- **FR-005**: Диагностическое логирование о рассогласовании роли в БД MAY быть сохранено как наблюдаемость, но MUST NOT влиять на возвращаемый результат.
- **FR-006**: Изменения MUST быть предварены характеризационными/репро-тестами (R-PROC-3), фиксирующими текущее наблюдаемое поведение обеих затронутых функций до внесения правок.
- **FR-007**: Публичный контракт слоя прав (сигнатуры и семантика оставшихся функций) MUST остаться неизменным для всех прочих вызывающих сторон.

### Key Entities *(include if feature involves data)*

- **Проверка прямого доступа**: логический ответ на вопрос «имеет ли пользователь прямой доступ к конкретному топику». Единственный источник — одна функция слоя прав.
- **Статус суперадмина**: логический ответ «является ли пользователь суперадмином», определяемый авторитетным идентификатором из конфигурации.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: После правки в слое прав остаётся 0 неиспользуемых функций-дублей проверки прямого доступа (было 1).
- **SC-002**: 100% существующих тестов прав доступа проходят без изменений ожиданий (поведение сохранено).
- **SC-003**: Проверка суперадмина не содержит ни одной ветви, недостижимой по результату (результат не зависящей от условия) — 0 таких ветвей (было 1).
- **SC-004**: Характеризационные тесты, зафиксировавшие поведение до правки, остаются зелёными после правки — подтверждают отсутствие регрессии поведения.
- **SC-005**: Footprint изменений ограничен слоем прав и его тестами (≤ 3 исходных файла: слой прав БД, агрегатор БД, сервис прав) — без затрагивания хендлеров и UI.

## Assumptions

- Дублирующая функция проверки прямого доступа (`has_direct_access`) не имеет ни одного live-вызова — подтверждено поиском по кодовой базе (только импорт в агрегаторе БД). Выживает `can_write`.
- Авторитетным источником истины для статуса суперадмина является идентификатор из конфигурации (`ADMIN_ID`); проверка роли в БД в текущей реализации на результат не влияет и признаётся мёртвой.
- Сохранение диагностического предупреждения о рассогласовании роли желательно для наблюдаемости, но не является обязательным требованием к результату.
- Изменения затрагивают только слой прав (`database/permissions.py`, `database/db.py`, `services/permission_service.py`) и соответствующие тесты; поведение конечного бота для пользователей неизменно.
