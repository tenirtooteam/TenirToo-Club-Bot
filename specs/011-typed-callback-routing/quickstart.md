# Quickstart: валидация типизированного роутинга

**Feature**: 011-typed-callback-routing | **Phase**: 1

Как проверить, что фича работает. Реализация — в `tasks.md`; здесь — что запускать и что обязано получиться.

## Предусловия

- `venv` активен; все команды — через `venv/Scripts/python.exe` (`R-PROC-7`).
- Docker Desktop поднят (нужен semgrep-гейту). Гейт пробит демона: `skip` ≠ «зелено» — CI в проекте нет, поэтому прогон до коммита обязателен.

---

## 1. Репро-тесты дефектов (RED до миграции)

Ядро `R-PROC-3`: три подтверждённых дефекта (`research.md` §2) получают падающие воспроизводящие тесты **до** любой миграции.

~~~bash
venv/Scripts/python.exe -m pytest tests/test_services/test_callback_routing_defects.py -v
~~~

**Ожидаемо ДО миграции**: 3 группы падают.

| Тест | Утверждает | Падает потому что |
|---|---|---|
| DEF-1 | `mod_topic_groups_55_pg_3` → экран топика **55**, страница **3** | приходит `topic_id=3` (номер страницы) |
| DEF-2 | `topic_assign_pg_777_pg_2` → экран выбора топика для юзера **777**, страница **2** | маршрут не матчится → «Ошибка навигации» |
| DEF-3 | `topic_in_group_5_9` → карточка топика **5** в контексте группы **9** | приходит `topic_id=9, group_id=5` |

**Ожидаемо ПОСЛЕ миграции**: все зелёные.

## 2. Характеризационный слой (GREEN всегда)

Фиксирует соответствие «callback_data → экран» для маршрутов с корректным текущим поведением (CHAR-OK, D-3). Пишется на исходном коде, обязан быть зелёным на **каждом** шаге миграции.

~~~bash
venv/Scripts/python.exe -m pytest tests/test_services/test_callback_routing_characterization.py -v
~~~

**Ожидаемо**: зелёный до миграции, зелёный после, без правок самих тестов. Правка характеризационного теста в ходе миграции = сигнал незапланированного изменения поведения → стоп и разбор.

## 3. Контрактные инварианты

~~~bash
venv/Scripts/python.exe -m pytest tests/test_services/test_callback_contract.py -v
~~~

Проверяет ([contracts/callback-routes.md](contracts/callback-routes.md)):

| Инвариант | Утверждение |
|---|---|
| R-1 | Каждая фабрика из `callbacks.py` присутствует в реестре навигатора ровно один раз |
| R-2 | Префиксы уникальны |
| C-1 | Ни один постоянный маршрут не содержит `:` |
| P-1 | Каждая фабрика, передаваемая паджинатору, имеет поле `page` |
| FR-011 | `pack()` на максимальных ожидаемых значениях не поднимает `ValueError` (≤64 байт) |
| Roundtrip | Для каждой фабрики `unpack(pack(x)) == x` |

Тест на 64 байта — именно «не поднимает `pack()`», а не ручной подсчёт длины: лимит обеспечивает библиотека (`research.md` §1).

## 4. Защитное поведение (FR-008)

~~~bash
venv/Scripts/python.exe -m pytest tests/test_services/test_ui_fuzzer.py tests/test_journeys/test_callback_defense.py -v
~~~

Каждый вход обязан дать `logger.warning` + возврат в главное меню, **без** исключения наружу:

| Вход | Класс |
|---|---|
| `nonexistent_route:1` | Неизвестный префикс |
| `mod_topic_groups:abc:3` | Непроходящий тип → `ValidationError` |
| `mod_topic_groups:55` | Неверная арность → `TypeError` |
| `user_info_5` | Старый формат (кнопка из истории чата) |
| `mod_topic_groups:55:-1` | Отрицательная страница |

## 5. Гейты статики (`R-ARCH-8`, `R-PROC-10`, `R-PROC-11`)

~~~bash
venv/Scripts/python.exe -m pytest tests/test_services/test_import_lint.py \
                                  tests/test_services/test_ruff_lint.py \
                                  tests/test_services/test_semgrep_lint.py \
                                  tests/test_services/test_callback_static_guard.py -v
venv/Scripts/python.exe -m ruff check .
venv/Scripts/python.exe -m lint_imports
~~~

Эти четыре файла — реальные машинные гейты репозитория. `tests/test_architecture.py` **не существует**; AST-тестов в проекте нет, несмотря на формулировку конституции. Не ссылаться на них.

**Ожидаемо**: новых нарушений нет. Новый корневой модуль `callbacks.py` не должен ломать контракты import-linter — он не импортирует ни `services`, ни `handlers`, ни `database` (чистые объявления формата, листовой узел графа импортов).

## 7. Ручная проверка сценариев, закрытых фиксами

Дефекты DEF-1/2/3 воспроизводимы только на списках длиннее одной страницы (limit = 7).

| Сценарий | Шаги | Ожидаемо после фичи |
|---|---|---|
| DEF-1 | Панель модератора → топик с >7 группами → «След. ▶️» | Страница 2 **того же** топика (не топика с ID = 2) |
| DEF-2 | Админка → участник → выдать роль → топиков >7 → «След. ▶️» | Страница 2 списка топиков (не «⚠️ Ошибка навигации») |
| DEF-3 | Админка → группа → список топиков → выбрать топик | Карточка **выбранного** топика (не топика с ID = ID группы) |
