## 1. Механика `CallbackData` в aiogram 3.4.1

**Источник**: `venv/Lib/site-packages/aiogram/filters/callback_data.py` `[ПРОВЕРЕНО: чтение исходника + исполняемая проверка]`.

| Факт | Значение | Следствие для плана |
|---|---|---|
| Разделитель по умолчанию | `:` (не `_`) | Префиксы вида `mod_topic_groups` легальны: `_` не конфликтует с разделителем. Переименование маршрутов не требуется. |
| Разделитель в префиксе | Запрещён (`ValueError` при объявлении класса) | Префикс не может содержать `:`. Все текущие имена маршрутов подходят. |
| `pack()` при >64 байт | Поднимает `ValueError` | **FR-011 обеспечивается библиотекой на этапе сборки клавиатуры.** Тест обязан проверить, что на максимальных значениях `pack()` не поднимает — а не измерять длину вручную. |
| `unpack()` при неверном префиксе | `ValueError` | — |
| `unpack()` при неверной арности | `TypeError` | — |
| `unpack()` при непроходящем типе | `pydantic.ValidationError` | `issubclass(ValidationError, ValueError) is True` `[ПРОВЕРЕНО]` |
| **Итог по отказам `unpack()`** | `(TypeError, ValueError)` покрывает все три | Навигатор ловит ровно этот кортеж — как и сам aiogram (см. ниже). |
| `Factory.filter()` | Возвращает `CallbackQueryFilter` | Матчинг **точный**: `unpack()` + `except (TypeError, ValueError): return False`. Ни `startswith`, ни вхождения подстроки. Инжектит в хендлер kwarg с именем `callback_data`. |

**Решение D-1**: навигатор ловит `(TypeError, ValueError)` вокруг `unpack()` и уходит в существующий warning-fallback.
**Обоснование**: тот же кортеж, что использует `CallbackQueryFilter.__call__` в самом aiogram — поведение фильтра и навигатора совпадает по определению, расхождение «фильтр пропустил / навигатор упал» невозможно.

**Решение D-2**: значения по умолчанию у полей **не** сокращают формат на проводе.
**Обоснование** `[ПРОВЕРЕНО: `unpack` требует `len(parts) == len(names)`]`: `unpack('mod_topic_groups:55')` для фабрики с полями `(topic_id, page=1)` поднимает `TypeError`. `pack()` всегда сериализует все поля. Следствие: любой синтетический переход из хендлеров сообщений обязан паковать `page` явно; «короткой формы» маршрута не существует.

## 2. Подтверждённые дефекты текущего роутинга

Планировалась чистая замена механизма. Фактически №19 сидит на **трёх живых дефектах**, каждый — прямое следствие подстрочного/позиционного разбора. Все три воспроизведены исполняемо.

### DEF-1 — Пять страничных маршрутов принимают номер страницы за ID сущности

`services/ui_service.py:308` вычисляет `p = callback_data.split("_")` по **полной** строке, включая хвост `_pg_{n}`. Маршруты, извлекающие ID как `int(p[-1])`, получают номер страницы.

`[ПРОВЕРЕНО: исполняемая проверка на `mod_topic_groups_55_pg_3` → ID=3 вместо 55]`

| Маршрут | Извлечение | Результат на `..._55_pg_3` |
|---|---|---|
| `mod_topic_groups_{tid}` | `int(p[-1])` | `topic_id = 3` ❌ |
| `mod_gr_addlist_{tid}` | `int(p[-1])` | `topic_id = 3` ❌ |
| `mod_users_manage_{tid}` | `int(p[-1])` | `topic_id = 3` ❌ |
| `mod_topic_moderators_{tid}` | `int(p[-1])` | `topic_id = 3` ❌ |
| `group_topics_list_{gid}` | `int(p[-1])` | `group_id = 3` ❌ |
| `user_templates_manage_{uid}` | `int(p[3])` | `user_id = 55` ✅ |
| `tmpl_act_start_{act}_{gid}` | `int(p[4])` | `group_id = 55` ✅ |

**Ключевое наблюдение**: два корректных маршрута корректны именно потому, что кто-то зашил **позиционный индекс с начала** (`p[3]`, `p[4]`) — обходя пагинационный баг. Обход применён непоследовательно, к 2 маршрутам из 7. Это и есть цена позиционного разбора: исправление живёт в индексе-магическом-числе, а не в контракте.

### DEF-2 — Маршрут `topic_assign_pg_{uid}` не маршрутизируется вообще

Префикс маршрута сам содержит `_pg`. `callback_data.split("_pg_")[0]` съедает его.

`[ПРОВЕРЕНО: `topic_assign_pg_777_pg_2` → `cmd='topic_assign'`, `page=777`, `p[-1]='2'`]`

- `cmd` = `topic_assign` → `cmd.startswith("topic_assign_pg")` = **False** → падение в fallback «⚠️ Ошибка навигации».
- `page` = **777** (это `user_id`).
- `p[-1]` = **2** (это номер страницы, а не `user_id`).

Все три значения неверны одновременно. Пагинация экрана «Выбор топика для роли» (`keyboards/admin_kb.py:281`, точка входа `handlers/admin.py:88`) **не работает**: нажатие «След. ▶️» выбрасывает администратора в главное меню. Экран достижим только первой страницей — через прямой рендер `handlers/admin.py:394`, минуя навигатор.

### DEF-3 — `topic_in_group_*` меняет местами `topic_id` и `group_id`

Производитель `keyboards/admin_kb.py:79`: `f"topic_in_group_{t_id}_{group_id}"`.
Потребитель `services/ui_service.py:322-323`: `show_topic_detail(state, event, int(p[-1]), int(p[-2]))`.
Сигнатура `services/ui_service.py:446`: `show_topic_detail(state, event, topic_id, group_id=0)`.

`[ПРОВЕРЕНО: кнопка `topic_in_group_5_9` (t_id=5, group_id=9) → навигатор зовёт `show_topic_detail(topic_id=9, group_id=5)`]`

Аргументы инвертированы: `p[-1]` — это `group_id`, а уходит в параметр `topic_id`. Пользователь, открывший топик из списка топиков группы, видит карточку **другого топика** — того, чей ID совпал с ID группы. Компенсации ниже по стеку нет: `show_topic_detail` использует `topic_id` напрямую в `db.get_topic_name(topic_id)`.

### Наблюдение: корректный разбор уже написан — и не используется навигатором

`handlers/moderator.py:44-51`:

~~~python
def extract_topic_id_from_callback(callback) -> int:
    base_data = callback.data.split("_pg_")[0]   # ← сначала отрезать пагинацию
    parts = base_data.split("_")
    try:
        return int(parts[-1])
    except (ValueError, IndexError):
        return -1
~~~

Это **правильный** алгоритм (отрезать пагинацию → затем брать хвост), с защитой от исключений. Он существует в репозитории, применяется частью хендлеров модератора — и **не применяется навигатором**, который делает `split("_")` по полной строке (DEF-1). Два разбора одного формата, один верный, другой нет, бок о бок.

Это буквальная материализация тезиса «формат живёт в двух головах» из спеки (User Story 2) — и главный аргумент за FR-001.

## 3. Влияние на FR-012 (неизменность поведения)

**Конфликт**: FR-012 требует, чтобы наблюдаемое поведение не менялось. DEF-1/2/3 означают, что для 7 маршрутов текущее поведение — дефект, и типизация его неизбежно чинит.

**Решение D-3**: характеризационный слой расщепляется на два множества.

| Множество | Что фиксирует | Состояние до миграции | Состояние после |
|---|---|---|---|
| **CHAR-OK** | Маршруты с корректным текущим поведением | GREEN | GREEN (без изменений) |
| **CHAR-DEF** | DEF-1/2/3 — тесты утверждают **корректное** поведение | **RED** (репро-тесты, `R-PROC-3`) | GREEN (починено миграцией) |

**Обоснование**: `R-PROC-3` и Constitution IV требуют для бага сначала падающий воспроизводящий тест. Фиксировать дефект как эталон характеризационным тестом — значит узаконить его. CHAR-DEF пишется как RED-репро до миграции и зеленеет ею.

**Правка спеки — ВЫПОЛНЕНА** (Q-1, вариант A, утверждён Шэфом 2026-07-14): FR-012 явно исключает DEF-1/2/3; добавлены FR-015 (фикс DEF-1), FR-016 (фикс DEF-2), FR-017 (фикс DEF-3), FR-018 (обязательные RED-репро до миграции), SC-009 и сценарий 5 в User Story 1. №19 расширен с «рефактора» до «рефактор + 3 фикса».

## 4. Границы scope: какие семейства маршрутов берём

Инвентаризация всех производителей `[ПРОВЕРЕНО: извлечение всех литералов `callback_data=` из `keyboards/`, `services/`, `handlers/`]` выявила **два независимых семейства** параметризованных маршрутов с разными разделителями:

| Семейство | Формат | Кто разбирает | В навигаторе? |
|---|---|---|---|
| **Легаси-навигатор** (`user_info_`, `mod_*`, `group_topics_list_`, `topic_in_group_`, `tmpl_act_*`, `help:`) | `prefix_arg` | `generic_navigator` — подстроки + позиции | **Да** |
| **События/анонсы/даты** (`event_*:{id}`, `ann_join:{id}:{f}`, `date_*:{iso}`) | `prefix:arg` | Хендлеры напрямую: `callback.data.split(":")[1]` (`handlers/events.py` ×10, `handlers/announcements.py` ×3) | Нет |
| **Поиск** (`search_start_*`, `search_pick_*`, `search_pg_*`) | `prefix_arg` | Хендлеры напрямую: `parts[2]`, `parts[-1]`, `"_".join(parts[3:-1])` | Нет |

`[ПРОВЕРЕНО: `grep -rn "CallbackData" --include=*.py .` вне venv → 0 совпадений]` — паттерн в проекте не используется нигде; фича вводит его впервые.

**Решение D-4**: scope фичи 011 = **семейство легаси-навигатора** + `build_paginated_menu` и все его вызывающие. Семейства «события/анонсы/даты» и «поиск» — **вне scope**, отдельной фичей.

**Обоснование**:
1. №19 в роадмапе адресует именно `generic_navigator`; остальные семейства к нему не относятся и удвоили бы footprint, не приблизив цель.
2. Ценность FR-001 — **помаршрутная**: маршрут, мигрированный на обоих концах, получает единый источник правды независимо от того, мигрированы ли соседи. Частичный охват **не** оставляет «двух голов» ни для одного маршрута — при условии, что каждый взятый маршрут мигрирован целиком. Это то, что делает границу безопасной.
3. Семейства `event_*`/`ann_*`/`date_*` страдают тем же классом (позиционный `split(":")[1]`) и заслуживают той же миграции — но это отдельная, самостоятельно ценная работа.

**Оговорка**: `build_paginated_menu` обслуживает и `search`, и `user_topics`. Изменение его контракта (D-5) затрагивает всех 16 вызывающих, включая `search`. Поэтому `search`-пагинация попадает в scope **механически** — как вызывающая сторона паджинатора, — тогда как разбор `search_start_`/`search_pick_` в хендлерах остаётся вне scope.

## 5. Контракт пагинации

**Проблема**: `build_paginated_menu(..., callback_prefix: str)` получает непрозрачную строку и клеит `f"{callback_prefix}_pg_{page±1}"` (`keyboards/pagination_util.py:66,70`). Это и есть источник DEF-1 и DEF-2.

**Решение D-5**: паджинатор принимает **экземпляр** `CallbackData` с полем `page` вместо строкового префикса; стрелки строятся как `cb.model_copy(update={"page": n}).pack()`.

**Обоснование**: `model_copy(update=...)` — штатный pydantic v2 (проект уже на 2.5.3 `[ПРОВЕРЕНО]`). Строковая хирургия исчезает полностью: номер страницы становится полем (FR-005), а `PAGINATED_CMDS` — отдельный реестр имён в `ui_service.py:13` — устраняется, потому что страничность следует из наличия поля `page` у фабрики (SC-004 в спеке → FR-005).

**Альтернативы отклонены**:
- *Передавать фабрику + dict полей* — воспроизводит ту же нетипизированную передачу, только в другой упаковке.
- *Оставить строковый префикс, чинить только `split`* — не устраняет DEF-2 (префикс, содержащий `_pg`) и оставляет формат в двух головах.

## 6. Сигнатура навигатора vs `R-UI-3`

**Проблема**: `R-UI-3` называет сигнатуру `UIService.generic_navigator(state, event, callback_data)`. При `Factory.filter()` aiogram инжектит **распакованный объект** в kwarg `callback_data`. Что хендлер передаёт навигатору?

**Решение D-6**: навигатор принимает `str | CallbackData`.
- `CallbackData` → используется напрямую (без повторного разбора);
- `str` без `:` → постоянный маршрут, поиск в словаре констант;
- `str` с `:` → префикс = `value.split(":")[0]`, **точный** поиск в реестре → `unpack()` → при `(TypeError, ValueError)` fallback;
- ничего не найдено → существующий warning-fallback (`ui_service.py:387-388`).

**Обоснование соответствия `R-UI-3`**: правило фиксирует **имя** параметра и **обязанность** Defensive Routing, а не тип параметра. Имя `callback_data` сохраняется — оно же совпадает с именем kwarg, который инжектит aiogram. Навигатор остаётся единственной точкой входа переходов; Defensive Routing усиливается (точный поиск + типизированная валидация вместо цепочки `in`). Пересмотр `R-UI-3` не требуется.

**Побочная ценность**: синтетические переходы из хендлеров сообщений (`generic_navigator(state, message, f"mod_topic_select_{topic_id}")`, `handlers/moderator.py:118` и ещё 8 мест) становятся типизированными объектами — сегодня это ручная f-строка, то есть третья голова того же формата.

**Оговорка по границе `R-UI-3`**: если при ревью Шэф сочтёт, что миграция входных фильтров хендлеров на `Factory.filter()` (FR-010) выходит за букву `R-UI-3` — требуется Route C на правку правила **до** реализации, не молчаливое отклонение (зафиксировано в спеке, Assumptions).

## Q-1 — РЕШЕНО (2026-07-14, Шэф)

**Вопрос**: расширять ли №19 с «рефактор» до «рефактор + 3 подтверждённых фикса» (правка FR-012)?

**Решение: вариант A — расширить.** Спека приведена в соответствие (см. §3). Отклонённые альтернативы: (B) сохранить дефекты — потребовало бы намеренно воспроизвести «ID = номер страницы» поверх типизированного механизма, что противоречит Constitution IV и смыслу фичи; (C) отдельная фича на фиксы вперёд — чистое разделение в истории, но два прохода по тем же файлам.

**Следствие для реализации**: ни один дефект не требует отдельной заплатки. DEF-1 и DEF-2 закрываются сменой контракта паджинатора (D-5), DEF-3 — реестром навигатора (D-6). Фиксы — побочный продукт типизации, а не добавленная работа; в этом и состоял аргумент за A.
