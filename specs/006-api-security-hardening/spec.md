# Feature Specification: API Security Hardening (Фаза 1)

**Feature Branch**: `006-api-security-hardening`

**Created**: 2026-07-09

**Status**: Draft

**Input**: User description: "Безопасность и целостность API (Фаза 1). Закрыть реально эксплуатируемую authz-дыру и связанные дефекты веб-слоя и обработки колбэков — единый путь записи на поход с проверкой доступа к топику и статуса одобрения, anti-replay для Telegram WebApp, корректный глобальный обработчик ошибок FastAPI, defense-in-depth на бот-колбэках удаления и выдачи прав."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Единое правило прямой записи на поход (Priority: P1)

Участник записывается на поход по **прямым** каналам: кнопкой «Иду» под анонсом в группе, тем же анонсом в Личном кабинете (Mini App) и из общего списка походов в Личном кабинете. Сегодня общий список в Личном кабинете (дашборд) не проверяет **ни** статус одобрения похода, **ни** — где это применимо — доступ участника к топику; анонс-каналы проверяют доступ к топику, но не одобрение. В результате участник может записаться на поход, ещё не прошедший модерацию, а через дашборд — в обход любых проверок. Нужно единое правило допуска для всех прямых каналов.

При этом у похода в модели данных нет собственного топика: топик-контекст появляется только у записи через **анонс** (топик анонса). Общий список Личного кабинета топик-контекста не имеет — там правило сводится к «поход одобрен + валидная сессия участника».

> Запись из **карточки** похода в боте — это отдельная модель заявки с одобрением организатора (аудит), а не прямая запись; она в это правило **не** входит и не изменяется.

**Why this priority**: Это единственный из дефектов фазы, который реально эксплуатируется рядовым участником (у каждого есть валидная веб-сессия), напрямую нарушает основную функцию продукта — контроль доступа — и обходит процесс модерации. Максимальный ущерб при минимальной сложности атаки.

**Independent Test**: Прогнать три сценария записи (бот-карточка, анонс, веб) от участника без доступа к топику и на неодобренный поход; во всех трёх запись должна быть отклонена одинаково. Тестируется автономно и сразу закрывает дыру.

**Acceptance Scenarios**:

1. **Given** запись идёт через анонс, привязанный к закрытому топику, и участник не имеет доступа к этому топику, **When** участник пытается записаться (кнопка анонса в боте или анонс в Личном кабинете), **Then** запись отклоняется с ответом «доступ запрещён» и участник не добавляется в список.
2. **Given** поход не одобрен (на модерации), **When** участник пытается записаться напрямую (любой прямой канал: дашборд, анонс в боте, анонс в Личном кабинете), **Then** запись отклоняется как недоступная до одобрения.
3. **Given** поход одобрен и (для анонс-канала) участник имеет доступ к топику анонса, **When** участник записывается через любой прямой канал, **Then** запись проходит и результат идентичен во всех прямых каналах.
4. **Given** участник уже записан, **When** он повторяет запись через любой прямой канал, **Then** система не создаёт дубликат и сообщает актуальный статус.
5. **Given** участник открывает общий список походов в Личном кабинете (без топик-контекста), **When** он записывается на **одобренный** поход, **Then** запись проходит (топик-проверка не применяется, т.к. у похода нет собственного топика).

### User Story 2 - Устаревшая веб-сессия не даёт бессрочного доступа (Priority: P2)

Данные веб-сессии Telegram (initData) подписаны, но не имеют срока годности: перехваченная строка авторизации работает вечно. Нужно, чтобы сессия считалась недействительной по истечении настраиваемого срока, и пользователю потребовалось переоткрыть Личный кабинет.

**Why this priority**: Снижает окно и ценность украденной сессии. Важно для безопасности, но требует предварительной компрометации initData, поэтому ниже P1.

**Independent Test**: Подать корректно подписанную сессию с временной меткой старше заданного порога и убедиться, что запрос отклоняется; со свежей меткой — проходит.

**Acceptance Scenarios**:

1. **Given** корректно подписанная веб-сессия с меткой времени старше настроенного срока жизни, **When** выполняется любой защищённый веб-запрос, **Then** запрос отклоняется как недействительная сессия.
2. **Given** корректно подписанная веб-сессия в пределах срока жизни, **When** выполняется защищённый запрос, **Then** запрос обрабатывается штатно.
3. **Given** сессия без метки времени, **When** выполняется защищённый запрос, **Then** запрос отклоняется (метка обязательна).

---

### User Story 3 - Сбой в вебе возвращает корректный ответ, а не второй сбой (Priority: P2)

При необработанной ошибке в веб-слое глобальный обработчик сам завершается некорректно, из-за чего клиент вместо аккуратного «внутренняя ошибка» получает нештатный сбой. Нужно, чтобы любая необработанная ошибка приводила к корректному ответу об ошибке сервера.

**Why this priority**: Влияет на стабильность и наблюдаемость веб-части, но не является дырой безопасности и проявляется лишь на уже возникшей ошибке.

**Independent Test**: Спровоцировать необработанную ошибку в защищённом эндпоинте и убедиться, что клиент получает корректный ответ со статусом ошибки сервера, а факт ошибки залогирован.

**Acceptance Scenarios**:

1. **Given** эндпоинт бросает необработанное исключение, **When** приходит запрос, **Then** клиент получает корректно оформленный ответ со статусом 500 и понятным текстом, а исключение зафиксировано в логах.

---

### User Story 4 - Действия удаления и выдачи прав проверяют полномочия на сервере (Priority: P3)

Подтверждения деструктивных операций (удаление групп/топиков/пользователей/походов, отзыв ролей) и назначение модератора / выдача прямого доступа выполняются обработчиками, которые полагаются на то, что кнопка «доверенно досталась» только администратору, и не перепроверяют права в момент действия. Нужно, чтобы полномочия проверялись сервером непосредственно перед мутацией.

**Why this priority**: В текущем UI такие кнопки появляются только в личке администратора и не могут быть подделаны рядовым участником, поэтому это защита в глубину, а не закрытие активного эксплойта. Ценно как страховка от будущих регрессий (кнопка попала не туда, изменился поток) и как единообразие модели прав.

**Independent Test**: Вызвать обработчик подтверждённого удаления и обработчик выдачи прав от лица пользователя без соответствующих полномочий и убедиться, что мутация не выполняется, а действие отклоняется.

**Acceptance Scenarios**:

1. **Given** пользователь без прав администратора, **When** инициируется обработчик подтверждённого удаления сущности, **Then** удаление не выполняется и действие отклоняется.
2. **Given** пользователь без прав на управление топиком, **When** инициируется назначение модератора или выдача прямого доступа для этого топика, **Then** мутация не выполняется и действие отклоняется.
3. **Given** пользователь с надлежащими правами, **When** он выполняет то же действие, **Then** операция проходит как прежде (регрессии нет).

---

### Edge Cases

- Поход не найден (удалён) в момент записи через любой канал — система отвечает «не найдено», без создания записей.
- Топик мероприятия не настроен (Default Deny) — запись через прямые каналы отклоняется для всех, кроме тех, у кого есть явный доступ, в соответствии с общей моделью доступа.
- Метка времени сессии из будущего (перекос часов) — трактуется как валидная в пределах разумного допуска; сильно будущие метки не должны обходить проверку срока.
- Некорректный/повреждённый формат метки времени в сессии — сессия отклоняется.
- Отзыв роли / выдача прав, когда целевая сущность уже изменилась (роль уже снята, доступ уже есть) — идемпотентный, предсказуемый ответ без ошибки.
- Одновременная запись/отписка из двух каналов — итоговое состояние согласовано, дубликатов участия не возникает.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST применять единое правило допуска к **прямой** записи участника на поход во всех прямых каналах (кнопка анонса в боте, анонс и общий список в Личном кабинете). Где у канала есть топик-контекст (запись через анонс), участник допускается только при праве писать в топик анонса. Общий список Личного кабинета топик-контекста не имеет — там применяется только правило одобрения (FR-002) при наличии валидной сессии. Запись из карточки бота — модель заявки/аудита — под это правило не подпадает.
- **FR-002**: Система MUST запрещать прямую запись на неодобренный поход (`is_approved = 0`) через все прямые каналы (дашборд, анонс в боте, анонс в Личном кабинете); поток создания/модерации автором и администрацией сохраняется без изменений.
- **FR-003**: Система MUST инкапсулировать правило допуска к прямой записи в единой переиспользуемой точке, чтобы прямые каналы не содержали расходящихся реализаций проверки.
- **FR-004**: Веб-эндпоинт переключения участия MUST выполнять проверки FR-001 и FR-002 перед любой мутацией и отвечать отказом с корректным статусом при их невыполнении.
- **FR-005**: Система MUST отклонять веб-сессию Telegram, если её метка времени авторизации старше настраиваемого срока жизни.
- **FR-006**: Срок жизни веб-сессии MUST быть настраиваемым параметром конфигурации со значением по умолчанию порядка 24 часов.
- **FR-007**: Система MUST отклонять веб-сессию, в которой отсутствует метка времени авторизации.
- **FR-008**: Глобальный обработчик необработанных ошибок веб-слоя MUST возвращать корректно оформленный ответ со статусом 500 и залогировать исходное исключение с трассировкой.
- **FR-009**: Обработчики подтверждённых деструктивных операций (удаление групп/топиков/пользователей/походов, отзыв ролей) MUST проверять полномочия инициатора на сервере непосредственно перед выполнением мутации.
- **FR-010**: Обработчики назначения модератора и выдачи прямого доступа к топику MUST проверять права инициатора на управление соответствующим топиком на сервере перед выполнением мутации.
- **FR-011**: Все отказы по правам и доступу MUST быть наблюдаемыми в логах (кто, какое действие, по какой причине отклонён) без раскрытия секретов.
- **FR-012**: Каждая правка MUST сопровождаться сначала падающим воспроизводящим тестом (pytest), фиксирующим дефект до исправления (R-PROC-3).
- **FR-013**: Изменения MUST сохранять существующее поведение для правомочных пользователей во всех затронутых потоках (отсутствие функциональных регрессий).

### Key Entities *(include if feature involves data)*

- **Веб-сессия (Telegram initData)**: подписанные данные авторизации пользователя Mini App; ключевые атрибуты — идентификатор пользователя, подпись целостности, метка времени авторизации. Срок годности определяется меткой времени и настраиваемым сроком жизни.
- **Поход (мероприятие)**: обладает статусом одобрения (на модерации / одобрен) и привязкой к топику, определяющей право участника записываться.
- **Правило допуска к записи**: единая политика, связывающая пользователя, топик мероприятия и статус одобрения в решение «разрешить/запретить запись».
- **Полномочие инициатора действия**: роль/права пользователя, инициирующего деструктивную операцию или выдачу прав, проверяемые в момент действия.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% попыток записи на поход от участника без доступа к топику мероприятия отклоняются во всех трёх каналах (0 успешных обходов).
- **SC-002**: 100% попыток записи на неодобренный поход через прямые каналы отклоняются.
- **SC-003**: Правомочные участники по-прежнему записываются и отписываются во всех трёх каналах без новых отказов (0 ложных запретов в регрессионных сценариях).
- **SC-004**: Веб-сессия с меткой времени старше настроенного срока отклоняется в 100% случаев; свежая — принимается в 100% случаев.
- **SC-005**: Любая необработанная ошибка веб-слоя возвращает корректный ответ со статусом 500 (0 случаев вторичного сбоя обработчика) и фиксируется в логах.
- **SC-006**: Каждый из четырёх дефектов покрыт как минимум одним воспроизводящим тестом, который падал до исправления и проходит после.
- **SC-007**: Действия удаления и выдачи прав, инициированные пользователем без полномочий, не выполняют мутацию в 100% проверенных сценариев.

## Assumptions

- Изменений схемы БД не требуется; правки затрагивают преимущественно веб-слой, middleware и обработчики.
- Единая модель «право писать в топик» (уже используемая в боте и в роутере анонсов веба) принимается как источник истины для правила допуска к записи; веб-дашборд приводится к ней.
- Прямая запись на поход трактуется как эквивалент клика «Иду» и подчиняется единому правилу; она включает три канала: кнопку анонса в боте, анонс и общий список в Личном кабинете.
- У похода нет собственного топика в схеме; топик-контекст есть только у записи через анонс (топик анонса). Поэтому общий список Личного кабинета гейтится по одобрению и валидной сессии, а не по топику.
- Запись из карточки похода в боте — модель заявки с одобрением организатора (аудит), вне унификации прямой записи; в этой фиче не изменяется.
- Значение по умолчанию срока жизни веб-сессии ≈ 24 часа; точное значение выносится в конфигурацию и может быть уточнено без изменения объёма фичи.
- Небольшой допуск на перекос часов при проверке метки времени сессии приемлем и выбирается по отраслевой практике.
- В текущем UI кнопки деструктивных действий появляются только в личных сообщениях администратора; серверная проверка прав добавляется как защита в глубину, а не как реакция на активный эксплойт.
- Существующая инфраструктура тестов (изолированная БД через фикстуры, pytest) переиспользуется; новых внешних зависимостей не предполагается.
