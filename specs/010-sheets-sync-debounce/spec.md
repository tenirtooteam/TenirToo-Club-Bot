# Feature Specification: Sheets-синк — дебаунс и владение фоновой задачей

**Feature Branch**: `010-sheets-sync-debounce`

**Created**: 2026-07-14

**Status**: Draft

**Input**: User description: "№17 Sheets-синк: рефактор ManagementService._trigger_sheets_sync. asyncio.create_task без хранения ссылки (GC-гонка, проглоченные ошибки, «Task was destroyed» в тестах — корень тест-патча из 007); полный ре-экспорт с N+1 по ролям на каждую мутацию (spam). Утверждён вариант B из PA-1: per-mode debounce с коалесценцией + shutdown-flush + батч-фетч ролей. Вне scope: to_thread для db.* (гейт за профайлом), полноценный rate-limit (№18). Фасад database.db неприкосновенен; сигнатуры 77 call-site'ов не менять."

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Всплеск правок не плодит экспорты (Priority: P1)

Администратор клуба выполняет серию быстрых изменений подряд (например, массово раздаёт роли, редактирует несколько групп или участников события). Каждое изменение сейчас немедленно инициирует полную выгрузку соответствующего листа в Google Sheets. Вместо десятков полных выгрузок подряд система должна выполнить одну итоговую выгрузку на затронутый тип данных после того, как всплеск утих.

**Why this priority**: Это корневая ценность №17 — устранение «спама» полных ре-экспортов, который нагружает Google API и БД при каждой единичной мутации. Без коалесценции остальные улучшения косметические.

**Independent Test**: Смоделировать N последовательных триггеров синка одного типа (`mode`) в пределах окна коалесценции и убедиться, что фактическая выгрузка в Google Sheets произошла ровно один раз, а не N раз.

**Acceptance Scenarios**:

1. **Given** окно коалесценции активно, **When** для одного `mode` подряд приходит 10 триггеров быстрее окна, **Then** выполняется ровно одна выгрузка этого `mode` с самыми свежими данными.
2. **Given** приходят триггеры двух разных типов данных (например, `users` и `groups`), **When** оба попадают в своё окно, **Then** каждый тип выгружается независимо и ровно один раз.
3. **Given** триггеры одного `mode` приходят с интервалом больше окна коалесценции, **Then** каждый обрабатывается как отдельная выгрузка (коалесценция не «склеивает» несвязанные во времени изменения).

---

### User Story 2 - Фоновая синхронизация владеется и не теряет ошибки (Priority: P1)

Фоновая задача синхронизации должна принадлежать системе на всё время своего выполнения: её нельзя терять из-за сборки мусора, а любая её ошибка должна быть зафиксирована в логах, а не проглочена. В тестовой среде задача не должна оставлять предупреждений вида «Task was destroyed but it is pending».

**Why this priority**: Это корневой фикс проблемы, которую в фиче 007 обходили тест-патчем: незапомненная `create_task` могла быть собрана GC до завершения и терять исключения. Владение задачей — обязательное условие надёжности синка.

**Independent Test**: Запустить триггер синка и проверить, что (а) ссылка на задачу удерживается системой до её завершения, (б) исключение внутри задачи попадает в лог, (в) прогон тестов не порождает предупреждений о разрушенных/незавершённых задачах.

**Acceptance Scenarios**:

1. **Given** запущена фоновая синхронизация, **When** задача выполняется, **Then** система удерживает ссылку на неё до завершения и снимает её после (без утечки ссылок).
2. **Given** внутри задачи выброшено исключение, **When** задача завершается, **Then** ошибка зафиксирована в логах и не всплывает как необработанное исключение задачи.
3. **Given** прогон полного набора тестов, **When** тесты, затрагивающие синк, завершаются, **Then** отсутствуют предупреждения «Task was destroyed» / «coroutine was never awaited» по этой задаче.

---

### User Story 3 - Остановка бота не теряет последнюю выгрузку (Priority: P2)

При штатной остановке бота изменения, попавшие в окно коалесценции, но ещё не выгруженные, не должны теряться: перед завершением работы система обязана немедленно прогнать все отложенные (pending) выгрузки без ожидания истечения окон.

**Why this priority**: Без этого дебаунс создаёт окно «данные в БД, но ещё не в Sheets»; при остановке в этом окне последняя выгрузка теряется. Это единственный критический нюанс варианта B (по PA-1), закрывающий пробел жизненного цикла.

**Independent Test**: Инициировать триггер синка, затем — до истечения окна коалесценции — вызвать процедуру штатной остановки и убедиться, что отложенная выгрузка выполнена немедленно, а не отброшена.

**Acceptance Scenarios**:

1. **Given** есть отложенные выгрузки в активных окнах коалесценции, **When** запускается штатная остановка бота, **Then** все отложенные выгрузки выполняются немедленно до завершения остановки.
2. **Given** нет отложенных выгрузок, **When** запускается остановка, **Then** процедура flush завершается без ошибок и без лишних выгрузок.

---

### Edge Cases

- Триггер приходит в тот момент, когда предыдущая выгрузка того же `mode` уже выполняется (не в фазе ожидания, а в фазе экспорта): система должна гарантировать последующую выгрузку со свежими данными, не теряя изменение и не порождая гонки.
- Google Sheets временно недоступен и выгрузка падает: ошибка логируется, отложенное состояние не «залипает» так, чтобы блокировать будущие выгрузки того же `mode`.
- Триггер типа `event_participants` привязан к конкретному `entity_id`: коалесценция не должна смешивать выгрузки участников разных событий.
- Штатная остановка при уже выполняющейся (не ожидающей) задаче: flush должен корректно дождаться/учесть выполняющуюся выгрузку без зависания.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система MUST коалесцировать множественные триггеры синка одного типа данных, поступившие в пределах окна коалесценции, в единственную итоговую выгрузку этого типа.
- **FR-002**: Система MUST различать типы синка (как минимум `users`, `groups`, `events`, `event_participants`, а также агрегирующий `all`) и вести коалесценцию по каждому типу независимо; для `event_participants` идентичность включает конкретное событие (`entity_id`).
- **FR-003**: Система MUST удерживать ссылку на каждую активную фоновую задачу синхронизации на всё время её жизни и освобождать ссылку по завершении, исключая потерю задачи из-за сборки мусора.
- **FR-004**: Система MUST фиксировать в логах любую ошибку фоновой задачи синхронизации и не допускать её всплытия как необработанного исключения задачи.
- **FR-005**: При штатной остановке бота система MUST немедленно выполнить все отложенные выгрузки без ожидания истечения окон коалесценции.
- **FR-006**: Система MUST заменить получение ролей по одному пользователю в цикле (N+1) на единый пакетный запрос ролей для набора пользователей через фасад данных `database.db`.
- **FR-007**: Система MUST NOT изменять сигнатуру точки входа `_trigger_sheets_sync`, вызываемую из существующих мест (~77 call-site'ов остаются без изменений).
- **FR-008**: Все обращения к данным MUST проходить через фасад `database.db`; прямой доступ к низкоуровневым модулям БД запрещён (Facade Integrity).
- **FR-009**: Итоговая выгрузка коалесцированного окна MUST отражать самое свежее состояние данных на момент фактического экспорта (не устаревший снимок момента первого триггера).
- **FR-010**: Прогон существующего набора тестов MUST завершаться без предупреждений о разрушенных/незавершённых задачах, связанных с фоновым синком.

### Key Entities *(include if feature involves data)*

- **Задача синхронизации (sync task)**: фоновая единица работы, выполняющая выгрузку одного типа данных; принадлежит системе, имеет удерживаемую ссылку и жизненный цикл (ожидание окна → экспорт → завершение/снятие ссылки).
- **Отложенное состояние синка (pending sync state)**: набор ожидающих выгрузок, сгруппированных по ключу типа данных (для `event_participants` — с учётом события); служит основанием для коалесценции и для shutdown-flush.
- **Тип синка (`mode`)**: категория выгружаемых данных, определяющая целевой лист и границу независимой коалесценции.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: При N (N ≥ 2) последовательных триггерах одного типа в пределах окна коалесценции фактическое число выгрузок этого типа равно 1.
- **SC-002**: Доля фоновых задач синхронизации, потерянных сборкой мусора до завершения, равна 0.
- **SC-003**: 100% ошибок фоновой синхронизации отражаются в логах (ни одна не проглатывается молча).
- **SC-004**: При штатной остановке с непустым отложенным состоянием 100% отложенных выгрузок выполняются до завершения остановки (потерь последней выгрузки нет).
- **SC-005**: Число обращений к данным для получения ролей при выгрузке пользователей не зависит от количества пользователей линейно: один пакетный запрос вместо одного на пользователя (устранение N+1).
- **SC-006**: Прогон полного набора тестов не порождает предупреждений «Task was destroyed»/«pending task» по фоновому синку (0 предупреждений).
- **SC-007**: Число мест вызова синка, потребовавших изменения сигнатуры, равно 0.

## Assumptions

- Google Sheets — фоновое зеркало данных, а не источник истины; кратковременное окно «данные в БД, но ещё не в Sheets» (порядка окна коалесценции) допустимо.
- Окно коалесценции — короткое (порядка секунд), конкретное значение фиксируется на этапе планирования; выбор значения не меняет объём фичи.
- Перевод блокирующих вызовов `database.db` внутри фоновой задачи на неблокирующее исполнение (`to_thread`/async-драйвер) — ВНЕ scope: гейтится за профилированием так же, как Ф1/Ф2 в фиче 008 (№15).
- Полноценный rate-limit / обработка 429 при выгрузке — ВНЕ scope; это отдельный пункт роадмапа (№18).
- Штатная остановка бота выполняется через существующий механизм жизненного цикла диспетчера (доступен хук завершения); включение shutdown-flush в этот механизм допустимо и является минимальным изменением.
- Существующее поведение выгрузки (какие данные и на какие листы отправляются) сохраняется без изменений; меняется только когда и как часто запускается фоновая задача и как она владеется.
