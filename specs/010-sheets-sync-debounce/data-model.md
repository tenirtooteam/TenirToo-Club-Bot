# Phase 1 Data Model: Sheets-синк дебаунс (№17)

Фича не вводит новых persistent-сущностей БД. Ниже — рантайм-структуры состояния и один новый read-контракт фасада.

## Runtime state (in-memory, ManagementService)

### `_pending_syncs: dict[str, asyncio.Task]`
- **Назначение**: реестр активных owned-задач синхронизации; основа коалесценции и shutdown-flush.
- **Ключ (sync key)**:
  - `mode` для `users` / `groups` / `events` / `all`
  - `f"event_participants:{entity_id}"` для участников конкретного события
- **Значение**: `asyncio.Task`, удерживаемая ссылка на весь жизненный цикл.
- **Инварианты**:
  - На один ключ в реестре — не более одной задачи.
  - Задача удаляется из реестра ровно в своём `add_done_callback` (по завершении/отмене).
  - Новый триггер существующего ключа отменяет старую pending-задачу до создания новой.
- **Жизненный цикл задачи**: `создана → ожидание окна (sleep) → экспорт → done → ключ снят`. Ветка отмены: `создана → ожидание → cancelled(в фазе sleep) → ключ снят` (экспорт не выполнялся).

### `SHEETS_SYNC_DEBOUNCE_SECONDS: float = 2.0`
- Модульная константа окна коалесценции (см. research R2). Патчится в тестах.

## Behaviour contract: `ManagementService`

### `_trigger_sheets_sync(mode="all", entity_id=None)` — **сигнатура НЕ меняется** (FR-007)
- Синхронный планировщик: вычисляет sync key, отменяет прежнюю pending-задачу этого ключа (если есть), создаёт новую owned-задачу дебаунса, кладёт в `_pending_syncs`, вешает `add_done_callback`.
- Не выполняет экспорт напрямую — только планирует.

### `flush_pending_syncs()` — **NEW**, `async`
- Для каждой задачи реестра: прервать фазу ожидания и немедленно выполнить экспорт соответствующего ключа; дождаться завершения; очистить реестр.
- Пустой реестр → немедленный возврат без выгрузок.
- Регистрируется как `dp.shutdown` хук в `main.py`.

## Read contract: фасад данных

### `db.get_roles_for_users(user_ids: list[int]) -> dict[int, list[tuple]]` — **NEW**
- **Слой**: реализация в `database/roles.py`, реэкспорт в `database/db.py` (R-ARCH-1).
- **Вход**: список `user_id`.
- **Выход**: `{user_id: [(role_name, topic_id), ...], ...}`. Пользователи без ролей → пустой список (ключ присутствует либо отсутствует — зафиксировать в реализации и тесте; потребитель обрабатывает отсутствие как «нет ролей»).
- **Правило совместимости**: для `ADMIN_ID` синтезируется `('superadmin', None)`, если роли нет — идентично `get_user_roles` (сохранение поведения выгрузки).
- **Заменяет**: N+1-цикл `for u in users: get_user_roles(u[0])` в фазе экспорта пользователей.
