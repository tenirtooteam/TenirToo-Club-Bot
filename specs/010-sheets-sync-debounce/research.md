# Phase 0 Research: Sheets-синк дебаунс (№17)

## R1. Механизм дебаунса per-mode

**Decision**: Модульный реестр `_pending_syncs: dict[str, asyncio.Task]` в `ManagementService`, ключ = `mode` либо `f"event_participants:{entity_id}"`. Каждая задача: `await asyncio.sleep(SHEETS_SYNC_DEBOUNCE_SECONDS)` → выполнить экспорт. Новый триггер того же ключа: отменить старую pending-задачу (если она ещё в фазе ожидания) и создать новую. `add_done_callback` снимает ключ из реестра.

**Rationale**: Отмена+пересоздание sleep-таска — простейший коалесцирующий дебаунс на stdlib, без внешних зависимостей и без отдельного воркера (минимальный footprint, Core Philosophy #5). Задача всегда в реестре → GC не может её собрать (FR-003). Отмена в фазе sleep безопасна (`CancelledError` не доходит до экспорта).

**Alternatives considered**:
- `loop.call_later()` + флаг «нужно ли ещё раз» — работает, но ручное управление `TimerHandle` + отдельный запуск экспорта усложняет владение и логирование ошибок. Отклонено: больше кода, та же семантика.
- Очередь + persistent worker (вариант C из PA-1) — отклонён в PA-1: модификация core `loader.py`/`main.py` под подсистему, переинжиниринг для клубного масштаба.

## R2. Значение окна коалесценции

**Decision**: `SHEETS_SYNC_DEBOUNCE_SECONDS = 2.0` (модульная константа `ManagementService`, легко правится).

**Rationale**: Sheets — фоновое зеркало (Assumptions спеки), не источник истины. 2 c достаточно, чтобы склеить типичный всплеск (серия ролей/правок в UI-сессии администратора идёт быстрее 2 c между кликами), и мало настолько, что окно «БД≠Sheets» неощутимо. Значение не влияет на объём фичи (Assumptions).

**Alternatives considered**: 0.5 c — слишком коротко, часть всплесков не склеится; 10 c — заметная задержка зеркала, риск потери большего объёма при нештатном падении. Отклонены.

## R3. Отмена pending-задачи, уже вошедшей в фазу экспорта

**Decision**: Отменять только если задача ещё не начала экспорт. Простейшая реализация: при новом триггере всегда `task.cancel()` старой pending; экспорт-корутина оборачивает фазу экспорта так, что `CancelledError` во время `sleep` = штатная замена, а во время самого экспорта Python всё равно даст ему возможность завершиться в текущем `await`. Для гарантии «последующая выгрузка со свежими данными» (edge case спеки): если новый триггер пришёл в момент выполняющегося экспорта — `cancel()` не остановит уже стартовавший `await export_*`, и новый триггер создаст новую pending-задачу, которая выгрузит свежие данные после окна. Потери изменения нет.

**Rationale**: Данные читаются из БД в момент фактического экспорта (FR-009), поэтому «догоняющая» выгрузка после всплеска всегда отражает финальное состояние. Гонки на потерю данных нет: худший случай — одна лишняя выгрузка, что приемлемо (лучше лишней, чем потерянной).

**Alternatives considered**: строгий lock «одна выгрузка mode за раз» с ре-триггер-флагом — надёжнее от лишних выгрузок, но добавляет состояние и сложность; для фонового зеркала избыточно. Отклонено (Core Philosophy #4/#5).

## R4. Shutdown-flush

**Decision**: `async def flush_pending_syncs()` в `ManagementService`: для всех задач в реестре — отменить фазу ожидания и немедленно выполнить экспорт синхронно (`await` каждого экспортного вызова), затем очистить реестр. Регистрация: `dp.shutdown.register(ManagementService.flush_pending_syncs)` в `main.py` (aiogram 3 `Dispatcher.shutdown` — сигнал, вызываемый при остановке polling). **[НЕ ПРОВЕРЕНО: точная сигнатура `dp.shutdown.register` в установленной версии aiogram — уточнить на этапе implement по фактическому API; при отличии использовать эквивалентный shutdown-хук диспетчера.]**

**Rationale**: `main.py` уже собирает `dp` (middlewares, routers) — регистрация хука там минимальна и не трогает `loader.py`. flush выполняет отложенные экспорты немедленно (FR-005, SC-004). Пустой реестр → flush завершается без выгрузок (edge case).

**Alternatives considered**: `atexit`/сигналы ОС — не async-дружелюбно, не гарантирует event loop. Отклонено. Регистрация в `loader.py` — `loader.py` намеренно тонкий (только объекты bot/dp), сборка — в `main.py`.

## R5. Пакетный фетч ролей (устранение N+1)

**Decision**: Новая функция `get_roles_for_users(user_ids: list[int]) -> dict[int, list[tuple]]` в `database/roles.py`, экспорт через `database/db.py`. Один SQL с `WHERE ur.user_id IN (...)` + JOIN на `roles`, группировка результата по `user_id` в питоне. Синтез `superadmin` для `ADMIN_ID` сохраняется (как в `get_user_roles`), чтобы поведение выгрузки не изменилось.

**Rationale**: Заменяет цикл `for u in users: db.get_user_roles(u[0])` (N запросов) на 1 запрос (SC-005, дух R-DATA-10). Живёт за фасадом (R-ARCH-1). Существующий `get_user_roles` не трогаем — обратная совместимость 77 call-site'ов и прочих потребителей.

**Alternatives considered**: правка `get_user_roles` под приём списка — сломала бы сигнатуру и множество вызовов. Отклонено (FR-007, Core Philosophy #5). Кэш ролей — вне scope (профайл-гейт, как №15).

## R6. Тестовая стратегия (R-PROC-3 TDD, R-TEST-3)

**Decision**: `tests/test_sheets_sync_debounce.py`:
1. **Репро (RED первым)**: триггер синка → отсутствие предупреждения «Task was destroyed»; задача удерживается и её ошибка логируется (мок `logger.error`, ассерт по `args`/`kwargs`).
2. **Коалесценция**: N быстрых триггеров одного mode в окне → мок `export_*` вызван 1 раз (SC-001).
3. **Разные ключи независимы**: `users`+`groups` → каждый export вызван по разу; `event_participants` разных `entity_id` не смешиваются.
4. **Flush**: триггер → до истечения окна `await flush_pending_syncs()` → export вызван немедленно (SC-004); пустой реестр → flush без выгрузок и без ошибок.
5. **N+1 устранён**: выгрузка users дёргает `get_roles_for_users` один раз, а не `get_user_roles` в цикле.

**Rationale**: Покрывает все FR/SC. Для управления окном в тестах — патч `SHEETS_SYNC_DEBOUNCE_SECONDS` на малое значение или прямой вызов внутренней корутины без ожидания. Мок-ассерты — `args`+`kwargs` (R-TEST-3). Изолированная БД через фикстуры (R-TEST-1).

**Alternatives considered**: реальные таймеры с `asyncio.sleep` в тестах — флейки/медленно. Отклонено в пользу патча константы/детерминированного прогона.
