# Data Model — Broadcast Rate-Limiting (013)

Фича не вводит схему БД. Все сущности — **в памяти процесса** (сознательный выбор, D-6/D-7). Ниже —
их форма, инварианты и жизненный цикл.

## E-1. Задание массовой ЛС-рассылки (`send_to_users`)

- **Вход**: `bot`, `user_ids: list[int]`, `text: str`, `reply_markup=None`.
- **Обработка**: `set(user_ids)` (дедуп, ID уже `int` — R-DATA-9), по одному сообщению на уникального
  адресата через `_send_message_resilient`, `asyncio.sleep(BROADCAST_PACING_SECONDS)` между отправками.
- **Инвариант**: каждый уникальный адресат получает ≤ 1 сообщение; недоступный адресат не прерывает
  список; flood-wait соблюдается (пауза + повтор), затем список дорабатывается до конца.
- **Хранения нет** — задание эфемерно, живёт на стеке вызова.

## E-2. Батч упоминаний @all (`send_native_all`)

- **Источник**: `authorized_users = db.get_topic_authorized_users(topic_id)` — список кортежей
  `(user_id, first_name, last_name)`.
- **Разбиение**: чанки `authorized_users[i : i + MENTION_BATCH_SIZE]` для `i in range(0, N, SIZE)`.
- **Тело батча**: `sender_name` + текст + скрытые упоминания `<a href="tg://user?id=UID">&#8203;</a>`
  по каждому UID чанка; одно сообщение в топик через `_send_message_resilient`, пауза между батчами.
- **Инвариант**: объединение всех батчей покрывает всех авторизованных (FR-005); при `N % SIZE == 0`
  пустого хвостового батча не возникает; пустой список → 0 сообщений.

## E-3. Запись частоты @all (`_all_cooldown`)

- **Форма**: `dict[int, float]` — `sender_id → timestamp последнего успешного @all` (`time.time()`).
- **Чтение**: перед рассылкой — `now - last <= ALL_MENTION_COOLDOWN_SECONDS` ⇒ отказ (кроме суперадмина).
- **Запись**: только после успешной рассылки (модератором); суперадмин запись не создаёт и не проверяет.
- **Жизненный цикл**: в памяти процесса; рестарт обнуляет (D-6). Рост ограничен числом отправителей
  (единицы-десятки модераторов) — потолок не требуется.
- **Изоляция тестов**: очищается `reset_notification_state()`.

## E-4. Запись анти-спам-алерта (`_alert_cache`)

- **Форма** (без изменений по ключу): `dict[tuple, float]` —
  `(user_id, topic_name) → ts` для default-deny (окно `DEFAULT_DENY_WINDOW_SECONDS`),
  `(user_id, "member_"+topic_name) → ts` для member-deny (окно `MEMBER_DENY_WINDOW_SECONDS`).
- **Дедуп-семантика (неизменна, FR-013)**: `now - last < window` ⇒ алерт подавляется; иначе —
  отправляется и `ts` обновляется.
- **Новое — ограничение роста (FR-012)**: `_prune_alert_cache(now)` в начале каждого алерт-метода
  удаляет записи с `now - ts > ALERT_CACHE_TTL_SECONDS`; при `len > ALERT_CACHE_MAX_ENTRIES` —
  вытеснение самых старых до потолка.
- **Инвариант**: размер ≈ число пар, виденных за последнее окно TTL, а не за всю жизнь процесса;
  подавление в пределах окна и повторная отправка после окна работают как прежде.
- **Изоляция тестов**: очищается `reset_notification_state()`.

## Именованные константы (FR-015)

| Константа | Назначение | Значение по умолчанию |
|---|---|---|
| `BROADCAST_PACING_SECONDS` | пауза между отправками/батчами | `0.05` |
| `FLOOD_WAIT_CAP_SECONDS` | потолок паузы по 429 | `30` |
| `FLOOD_WAIT_MAX_RETRIES` | повторов после flood-wait | `1` |
| `MENTION_BATCH_SIZE` | упоминаний на одно @all-сообщение | `50` |
| `ALL_MENTION_COOLDOWN_SECONDS` | интервал @all на отправителя | `60` |
| `DEFAULT_DENY_WINDOW_SECONDS` | окно дедупа default-deny алерта | `60` |
| `MEMBER_DENY_WINDOW_SECONDS` | окно дедупа member-deny алерта | `3600` |
| `ALERT_CACHE_TTL_SECONDS` | порог очистки протухших записей кэша | `3600` |
| `ALERT_CACHE_MAX_ENTRIES` | жёсткий потолок размера кэша | `1000` |
