# Data Model: Персистентный FSM-storage (feature 012)

**Дата**: 2026-07-17 | **Спека**: [spec.md](spec.md) | **Решения**: [research.md](research.md)

## Таблица `fsm_storage`

Создаётся в существующем потоке `init_db()` (`database/connection.py`), идемпотентно
(`CREATE TABLE IF NOT EXISTS`), рядом с остальными таблицами — FR-007. На работающих базах
добавляется без миграции: таблица новая, существующих данных не касается.

~~~sql
CREATE TABLE IF NOT EXISTS fsm_storage (
    bot_id     INTEGER NOT NULL,
    chat_id    INTEGER NOT NULL,
    user_id    INTEGER NOT NULL,
    thread_id  INTEGER NOT NULL DEFAULT 0,
    destiny    TEXT    NOT NULL DEFAULT 'default',
    state      TEXT,
    data       TEXT    NOT NULL DEFAULT '{}',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    PRIMARY KEY (bot_id, chat_id, user_id, thread_id, destiny)
)
~~~

### Столбцы

| Столбец | Тип | Смысл |
|---|---|---|
| `bot_id`, `chat_id`, `user_id` | INTEGER NOT NULL | Составляющие `StorageKey` aiogram. |
| `thread_id` | INTEGER NOT NULL DEFAULT 0 | `StorageKey.thread_id`; **`None` → `0`** (см. инвариант I-1). |
| `destiny` | TEXT NOT NULL DEFAULT 'default' | `StorageKey.destiny`; aiogram по умолчанию шлёт `"default"`. |
| `state` | TEXT (nullable) | Имя состояния (`State.state`) либо `NULL` — состояние снято. |
| `data` | TEXT NOT NULL DEFAULT '{}' | JSON-объект данных FSM (D-3). Пустые данные — `'{}'`, не `NULL`. |
| `updated_at` | TIMESTAMP | Пассивные метаданные (FR-012). Заполняется SQL-ом, не Python (D-7). |

**Внешних ключей нет** — осознанно (D-2): состояние существует и у незарегистрированного
пользователя, `chat_id` не пользователь, а `ON DELETE CASCADE` тихо стирал бы состояния.
`PRAGMA foreign_keys=ON` при этом остаётся в силе на соединении (`R-DATA-2`) — у этой таблицы
просто нет FK.

**Индексов, кроме PK, нет**: единственный шаблон доступа — точечный поиск по полному ключу,
который PK и обслуживает. Объём O(число пользователей) ≈ сотни строк.

## Инварианты

- **I-1 (единственность записи на владельца)**: для одного `StorageKey` в таблице существует
  **ровно одна** строка. Обеспечивается составным PK **и** сентинелом `thread_id = 0`.
  *Почему сентинел обязателен*: SQLite считает NULL-ы в UNIQUE-индексе различными, поэтому
  `thread_id NULL` допустил бы дубликаты и обошёл `ON CONFLICT`; `thread_id=None` — основной
  случай (личные сообщения). Проверено пробом — [research.md](research.md) §2. Запирается
  тестом: две записи одним ключом при `thread_id=None` → `SELECT count(*) == 1`.
- **I-2 (семантика пустоты)**: отсутствие строки эквивалентно `state = None`, `data = {}`.
  Чтение неизвестного ключа возвращает именно это и не падает — совпадает с `MemoryStorage`
  `[VERIFIED]`.
- **I-3 (сохранность типов)**: значение, записанное в `data`, читается обратно тем же типом:
  `list[int]` остаётся `list[int]`, `bool` остаётся `bool`, `None` остаётся `None` (FR-004).
  Границы JSON (`tuple → list`, нестроковый ключ → строка) сегодня недостижимы — инвентарь
  значений это подтверждает ([research.md](research.md) §3) — и заперты round-trip тестом.
- **I-4 (порог удаления)**: строка удаляется **только** при `state IS NULL` **и** `data == {}`
  одновременно. Удаление по одному лишь снятому состоянию запрещено — оно уничтожило бы
  трекинг-ключи стерильного UI и нарушило бы `R-FSM-1` (D-4).
- **I-5 (приватность схемы)**: `fsm_storage` не появляется ни в одной доменной операции фасада
  `database/db.py`. Через фасад ре-экспортируется только класс `SQLiteStorage` — конструктор
  для проводки в `loader.py`, не доступ к данным (D-1).
- **I-6 (изоляция владельцев)**: состояния разных `(bot_id, chat_id, user_id)` не пересекаются;
  чтение по ключу не может вернуть чужую запись (FR-002).

## Отображение `StorageKey` → строка

| `StorageKey` | Столбец | Преобразование |
|---|---|---|
| `bot_id` | `bot_id` | как есть |
| `chat_id` | `chat_id` | как есть |
| `user_id` | `user_id` | как есть |
| `thread_id: Optional[int]` | `thread_id` | `None → 0` (I-1) |
| `destiny: str` | `destiny` | как есть |

Обратное преобразование (`0 → None`) не требуется: `StorageKey` наружу не отдаётся — storage
только принимает ключ, но никогда его не конструирует.

## Жизненный цикл записи

- **нет строки → строка**: первый `set_state`/`set_data` с непустым содержимым (UPSERT).
- **строка → строка**: любой последующий `set_state`/`set_data` — UPSERT по PK, `updated_at`
  переставляется. Последняя запись выигрывает (закрывает edge case конкурентной записи —
  «рваного» полусохранённого состояния не бывает, UPSERT атомарен).
- **строка → нет строки**: `set_state(None)`/`set_data({})`, приведшие к состоянию I-4.
- **переживание рестарта**: строка не зависит от жизни процесса — в этом вся фича. Новый
  процесс (новый экземпляр `SQLiteStorage` поверх того же файла) читает её по тому же ключу.
- **битая строка**: нечитаемый JSON в `data` трактуется как `data = {}` + запись в лог, строка
  не удаляется автоматически (FR-009). Обработка события продолжается.

## Сущности спеки → реализация

| Сущность спеки | Здесь |
|---|---|
| Запись состояния (state record) | строка `fsm_storage` |
| Ключ адресации (storage key) | составной PK, `StorageKey` aiogram |
| Трекинг-ключи стерильного UI | обычные ключи внутри JSON `data`; storage их не различает (FR-003) — но именно I-4 не даёт их потерять |
