# Quickstart: как проверить feature 012

**Дата**: 2026-07-17 | **Спека**: [spec.md](spec.md) | **Контракт**: [contracts/storage-contract.md](contracts/storage-contract.md)

## 0. Предусловия

- `venv` активен — глобальный Python без pytest (`R-PROC-7`).
- Docker поднят: semgrep-гейт без живого демона отдаёт `skip`, а **`skip` ≠ «зелено»**, CI нет.

## 1. Полный прогон

~~~
venv\Scripts\python.exe -m pytest
~~~

Ожидание: зелено целиком. Ключевой сигнал SC-006 — **существующий корпус тестов не
редактировался**: замена storage не должна требовать ни одной правки в местах вызова FSM.
Понадобилась правка существующего теста — это не «поправить тест», а сигнал остановиться и
разобраться, что за поведение поехало.

## 2. Тесты фичи

~~~
venv\Scripts\python.exe -m pytest tests/test_database/test_fsm_storage.py -v
~~~

Что обязано быть закрыто (соответствие требованиям — в [contracts/storage-contract.md](contracts/storage-contract.md) §4):

| Проверка | Требование |
|---|---|
| RED-репро: `MemoryStorage` теряет состояние при «рестарте», `SQLiteStorage` — нет | `R-PROC-3`, FR-001 |
| Round-trip по реальному инвентарю ключей, типы сохранены | FR-004, I-3 |
| `thread_id=None`, две записи одним ключом → ровно одна строка | I-1 |
| `set_state(None)` при непустых данных строку **не** удаляет | `R-FSM-1`, I-4 |
| `state=None` + `data={}` → строки нет | FR-010, I-4 |
| Неизвестный ключ: `get_state() → None`, `get_data() → {}` | I-2 |
| Мусор в `data` → `{}` + лог, без исключения | FR-009 |
| Разные владельцы не пересекаются | FR-002, I-6 |
| `close()` не закрывает общее соединение | D-5 |

**Как эмулируется рестарт**: создаётся **новый экземпляр** `SQLiteStorage` поверх той же
временной БД. Честная модель — в проде новый процесс это ровно новый экземпляр над тем же
файлом. Никаких `subprocess`.

## 5. Живая проверка (после мержа, руками)

Автотесты доходят до предела, достижимого без живого Telegram. Реальный рестарт процесса они
не воспроизводят — только его модель. Прокликать обязательно:

1. **Мусор**: в ЛС открыть любое меню → **перезапустить бота** → нажать кнопку в этом меню
   (или `/start`) → предыдущее меню обязано удалиться, а не остаться висеть.
2. **Ввод**: начать переименование топика (бот просит «Введите название») → **перезапустить
   бота** → прислать текст → топик обязан переименоваться. До фичи текст уходил в пустоту.
3. **Онбординг**: пройти онбординг админа → **перезапустить бота** → открыть админ-панель →
   приветственного экрана быть не должно.
4. **Изоляция**: два разных аккаунта одновременно в разных состояниях → рестарт → каждый
   продолжает своё, состояния не перепутаны.
5. **Чистая БД**: удалить `database/bot.db` → запустить бота → таблица создаётся, бот
   поднимается без ошибок (FR-007).
6. **Существующая БД**: запустить фичу на базе, созданной до неё → данные на месте, таблица
   добавилась, ручная миграция не потребовалась (FR-007).

## 6. Проверка глазами в БД

~~~
venv\Scripts\python.exe -c "import sqlite3; c=sqlite3.connect('database/bot.db'); print(c.execute('SELECT bot_id,chat_id,user_id,thread_id,destiny,state,data,updated_at FROM fsm_storage').fetchall())"
~~~

Ожидание: `data` — читаемый JSON; на одного владельца ровно одна строка; `thread_id = 0` для
личных сообщений (сентинел, не NULL — см. I-1).
