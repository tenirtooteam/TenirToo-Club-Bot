## §1. Контракт aiogram 3.4.1 — что именно надо реализовать

`[VERIFIED: venv/Lib/site-packages/aiogram/fsm/storage/base.py]`

`BaseStorage` — ABC с **пятью** абстрактными методами:

| Метод | Сигнатура | Обязателен |
|---|---|---|
| `set_state` | `(key: StorageKey, state: StateType = None) -> None` | да |
| `get_state` | `(key: StorageKey) -> Optional[str]` | да |
| `set_data` | `(key: StorageKey, data: Dict[str, Any]) -> None` | да |
| `get_data` | `(key: StorageKey) -> Dict[str, Any]` | да |
| `close` | `() -> None` | да |
| `update_data` | `(key, data) -> Dict[str, Any]` | **нет** — есть дефолтная реализация |

`StateType = Optional[Union[str, State]]` — `set_state` обязан принимать и объект `State`, и
строку, и `None`. Эталон (`MemoryStorage`) разворачивает так:
`state.state if isinstance(state, State) else state` `[VERIFIED: fsm/storage/memory.py:41]`.

`StorageKey` — `frozen dataclass` с полями: `bot_id: int`, `chat_id: int`, `user_id: int`,
`thread_id: Optional[int] = None`, `destiny: str = "default"`. **Поля
`business_connection_id` в 3.4.1 нет** `[VERIFIED]` — оно появилось в более поздних версиях;
схему под него не закладываем (иначе апгрейд aiogram из Фазы 4 придётся обсуждать здесь).

**Семантика отсутствующей записи** `[VERIFIED: memory.py:33 — defaultdict]`: `MemoryStorage`
на неизвестном ключе возвращает `get_state() -> None` и `get_data() -> {}`, не падая. Наш
storage обязан вести себя так же — это часть контракта, а не мелочь.

**`close()` реально вызывается** `[VERIFIED: fsm/middleware.py:110]`:
`FSMContextMiddleware.close()` делает `await self.storage.close()` при остановке диспетчера.
Это диктует D-5.

## §2. Ключевая находка — NULL в составном первичном ключе SQLite

**Проб** (`venv\Scripts\python.exe`, sqlite 3.45.1) `[VERIFIED]`:

~~~
CREATE TABLE t (a INTEGER NOT NULL, b INTEGER, PRIMARY KEY (a,b));
INSERT INTO t VALUES (1,NULL);
INSERT INTO t VALUES (1,NULL);
SELECT count(*) FROM t;  -->  2
~~~

SQLite допускает NULL в столбцах составного PK и считает NULL-ы **различными** (составной PK
реализован как UNIQUE-индекс, а в нём NULL ≠ NULL). Дубликат проходит, `ON CONFLICT` не
срабатывает.

**Почему это критично именно здесь**: `thread_id` в `StorageKey` по умолчанию `None`, и это
не редкий угол, а **основной путь** — все личные сообщения с ботом. Схема с
`thread_id INTEGER` и PK «как есть» дала бы новую строку на каждую запись состояния каждого
пользователя в ЛС, а чтение (`SELECT ... LIMIT 1`) стало бы недетерминированным. Фича была бы
сломана ровно на главном сценарии, причём тихо.

**Решение (D-2)**: `thread_id INTEGER NOT NULL DEFAULT 0`, `None → 0` на входе в storage,
`0 → None` не нужен (наружу ключ не отдаём). Сентинел `0` безопасен: `thread_id` — это
`message_thread_id` Telegram, то есть ID сообщения, всегда > 0. Инвариант закрывается тестом
(конфликт-тест: две записи одним ключом с `thread_id=None` → ровно одна строка).

**UPSERT доступен** `[VERIFIED: sqlite 3.45.1 ≫ 3.24]` — `INSERT ... ON CONFLICT DO UPDATE`
законен, отдельный `SELECT`-then-`INSERT/UPDATE` (с гонкой) не нужен.

## §3. Инвентарь значений FSM — можно ли JSON

`[VERIFIED: grep по `update_data(`/`set_data(` вне venv — 19 продуктовых мест]`

| Ключ | Тип значения |
|---|---|
| `last_menu_ids` | `list[int]` (в т.ч. `[]`) |
| `last_menu_id` | `int` / `None` |
| `admin_onboarded` | `bool` |
| `edit_topic_id`, `edit_group_id`, `edit_user_id`, `edit_event_id` | `int` |
| `moderator_edit_topic_id`, `moderator_direct_access_topic` | `int` |
| `search_type`, `search_action`, `search_query` | `str` |
| `search_context` | `int` |
| `title`, `new_title`, `dates` | `str` |
| `start_iso`, `end_iso` | `str` / `None` |

**Все значения JSON-safe**: `None` / `int` / `str` / `bool` / `list[int]`. Ни `datetime`, ни
`tuple`, ни `set`, ни объектов aiogram, ни нестроковых ключей словаря.

**Решение (D-3)**: JSON, не pickle. Обоснование: (а) инвентарь подтверждает достаточность;
(б) pickle исполняет произвольный код при десериализации — держать такое в БД, куда пишет
пользовательский ввод (`title`, `search_query`), не нужно даром; (в) JSON читаем глазами при
отладке; (г) `R-CODE`-дух: простое решение без нового механизма.

**Известная граница JSON (FR-004)**: `tuple → list`, нестроковый ключ словаря → строка.
Сегодня таких значений нет, но контракт нужно **запереть тестом round-trip по инвентарю**, а
не комментарием: иначе первый, кто положит в FSM кортеж, узнает об этом в проде.

## §4. Решения

### D-1. Где живёт модуль

- **Решение**: `database/fsm_storage.py` — внутренний модуль пакета `database`, импортирует
  `.connection` напрямую, как это делают его соседи (`members.py`, `topics.py`…). Класс
  `SQLiteStorage` ре-экспортируется через фасад `database/db.py`; `loader.py` берёт его
  **через фасад** (`from database.db import SQLiteStorage`).
- **Обоснование**: это персистентность — ей место в `database/`. `R-ARCH-1` запрещает
  внешним потребителям импортировать внутренние DB-модули в обход фасада, поэтому `loader.py`
  идёт через `db.py`. Ре-экспорт **класса** — не доменный доступ к данным: ни один сервис не
  получает функций чтения/записи FSM, у фасада не появляется ни одной новой операции над
  состояниями. Инвариант «схема FSM — приватная деталь» сохраняется.
- **Альтернативы**: (1) корневой `fsm_storage.py` рядом с `callbacks.py`/`config.py`,
  использующий `db.get_conn()` — работает, но кладёт persistence-модуль вне `database/`
  вопреки смыслу слоя; отвергнут. (2) `loader.py → database.connection` напрямую — прямое
  нарушение `R-ARCH-1` ради экономии одной строки; отвергнут.
- **Побочный эффект**: появляется новая стрелка графа импортов `loader.py → database/db.py`.
  Граф описан в `docs/knowledge/architecture.md` (§Import Dependency Graph) — требуется
  обновление по Route C **в том же коммите**. import-linter это не поймает: его
  `root_packages` — `handlers`/`middlewares`/`services`/`database`, корневые модули в контракт
  не входят `[VERIFIED: .importlinter]`.

### D-2. Схема и ключ

- **Решение**: см. [data-model.md](data-model.md). Составной PK
  `(bot_id, chat_id, user_id, thread_id, destiny)`, все столбцы ключа `NOT NULL`,
  `thread_id` с сентинелом `0` вместо `None` (§2).
- **Без FOREIGN KEY на `users(user_id)`**. Обоснование: (а) FSM-состояние существует и у
  незарегистрированного пользователя — запись в `users` появляется позже, в
  `UserManagerMiddleware`; FK ронял бы запись состояния на первом же контакте; (б) `chat_id`
  вообще не пользователь; (в) `ON DELETE CASCADE` тихо стирал бы состояния при удалении
  участника; (г) таблица инфраструктурная, а не доменная — у неё нет референциальных
  отношений с доменом, и их отсутствие — часть аргумента «это не часть фасада».
  `R-DATA-2` (FK ON на соединении) при этом не нарушается: PRAGMA остаётся включённой,
  просто у этой таблицы нет внешних ключей.

### D-3. Сериализация — JSON

См. §3. Дополнительно: **ошибка сериализации при записи не глушится**. `FR-009`
(деградировать до «нет состояния») относится к **чтению** битой записи. На записи
`TypeError` от `json.dumps` обязан всплыть громко: проглотить его — значит молча потерять
состояние, то есть воспроизвести ровно тот дефект, который чиним.

### D-4. Когда удалять строку (FR-010) — и почему это опасное место

- **Решение**: строка удаляется **только если одновременно** `state IS NULL` **и** `data == {}`.
- **Обоснование — тут закопана мина `R-FSM-1`**. Соблазн «`set_state(None)` → удалить строку»
  выглядит естественно и **уничтожил бы стерильный интерфейс**: штатная связка проекта —
  `state.set_state(None)` + `clear_fsm_data_safely`, которая сохраняет `last_menu_ids`,
  `last_menu_id`, `admin_onboarded` `[VERIFIED: services/ui_service.py:71-81]`. Удаление
  строки по `set_state(None)` стёрло бы трекинг-ключи и дало бы неубираемый мусор — то самое
  следствие №16, ради устранения которого затеяна фича, только теперь встроенное в storage
  намеренно. Правило `R-FSM-1` запрещает `state.clear()` именно за этот сценарий; storage
  обязан уважать ту же границу.
- **Практика**: удаление будет срабатывать редко (после `clear_fsm_data_safely` словарь почти
  всегда непустой). Это нормально: FR-010 — про «не копить пустые записи», а не про агрессивную
  зачистку. Объём и так O(пользователей).

### D-5. `close()` — no-op

- **Решение**: `async def close(self) -> None: pass`.
- **Обоснование**: соединением владеет `database/connection.py` (`_shared_conn`,
  `close_shared_conn()` — единственная точка закрытия). Диспетчер зовёт `storage.close()` на
  остановке `[VERIFIED: fsm/middleware.py:110]`; закрой мы там общее соединение — storage
  дёрнул бы из-под ног соединение, которым пользуется весь остальной код (в т.ч. FastAPI-часть,
  живущая в том же процессе через `asyncio.gather`). Владение остаётся у владельца.

### D-6. `update_data` — наследуем дефолт

- **Решение**: не переопределять; работает унаследованная реализация (`get_data` → `dict.update`
  → `set_data`).
- **Обоснование**: точное совпадение семантики с `MemoryStorage` бесплатно. Цена — 2 обращения
  к БД вместо одного на вызов; при 200 пользователях и локальном SQLite это не обсуждается.
  Переопределение ради экономии — преждевременная оптимизация с риском разойтись с эталоном.

### D-7. `updated_at` — через SQL, не через Python

- **Решение**: заполняется в SQL (`CURRENT_TIMESTAMP`), а не `datetime.now()` в Python.
- **Обоснование**: FR-012 делает метку пассивной (ни одно решение от неё не зависит), а
  `CURRENT_TIMESTAMP` в SQLite — UTC. Это **обходит долг Фазы 4** (наивный `datetime.now()`
  без таймзоны по всему проекту): фича не добавляет ни одной новой наивной метки времени.
  Остальные таблицы (`audit_requests`, `announcements`) уже используют
  `DEFAULT CURRENT_TIMESTAMP` `[VERIFIED: database/connection.py:184,198]` — паттерн проекта
  соблюдён.

### D-8. Порядок инициализации — почему конструктор не трогает БД

- **Факт** `[VERIFIED: loader.py + main.py:49]`: `loader.py` создаёт `dp = Dispatcher(storage=...)`
  на **импорте модуля**, а `db.init_db()` вызывается позже, внутри `main()`. На момент
  конструирования storage таблицы может не быть.
- **Решение**: конструктор `SQLiteStorage` не выполняет ни одного обращения к БД (ни
  `CREATE TABLE`, ни `get_conn()`). Таблица создаётся в существующем потоке `init_db()`
  (FR-007), а storage касается БД только в момент обработки события — заведомо после
  `init_db()`. Ленивое разрешение соединения через `get_conn()` даёт это бесплатно.
- **Бонус для тестов**: storage не кэширует ни соединение, ни путь к БД, поэтому автоюзная
  фикстура `db_setup` (подменяет `connection.DB_PATH` + зовёт `init_db()` на каждый тест
  `[VERIFIED: tests/conftest.py:26-37]`) прозрачно уводит и storage на временную БД. **Своя
  reset-фикстура не нужна** — в отличие от `reset_registration_cache`/`reset_sheets_sync_state`,
  у storage нет глобального состояния в процессе. Это упрощение относительно исходной
  установки на планирование, и оно обосновано: сбрасывать нечего.

### D-9. Изоляция событий — не трогаем

- **Решение**: `events_isolation` остаётся дефолтной (`DisabledEventIsolation`).
- **Обоснование**: вне scope. Персистентность состояния и сериализация параллельных апдейтов —
  разные задачи; `SimpleEventIsolation` — это про блокировки, а не про переживание рестарта.
  Edge case «две записи подряд» закрывается атомарностью UPSERT (последняя запись выигрывает),
  чего спека и требует.

### D-10. Тестовая стратегия — существующая фикстура `storage` остаётся `MemoryStorage`

- **Факт** `[VERIFIED: tests/conftest.py:80-83,103]`: фикстура `storage` отдаёт `MemoryStorage`,
  и на ней построены `create_context` / `create_callback`, то есть весь существующий корпус
  тестов хендлеров.
- **Решение**: не трогать. Тесты хендлеров проверяют хендлеры, им storage безразличен; замена
  на SQLite замедлила бы прогон и размыла бы предмет теста. Новый storage покрывается
  **отдельным** файлом тестов с собственной фикстурой.
- **Как тестируется «рестарт»**: рестарт процесса эмулируется созданием **нового экземпляра**
  `SQLiteStorage` поверх той же (временной) БД. Это честная модель: в проде новый процесс —
  это ровно новый экземпляр над тем же файлом. Никаких `subprocess`.
- **RED-first (`R-PROC-3`)**: №16 — дефект («рестарт теряет состояние»), значит первым пишется
  падающий тест, воспроизводящий потерю на `MemoryStorage`, и он же становится зелёным на
  `SQLiteStorage`.
