# Telegram Bot: Access Control and Community Management

## Обзор
Профессиональный модульный Telegram-бот на базе **aiogram 3**, разработанный для туристического клуба **«Теңир-Тоо»**. Система обеспечивает надёжное управление доступом к темам (топикам) форума внутри Telegram-супергруппы, автоматизацию учёта участников клуба и защиту интерфейса от ошибок двойных нажатий.

---

## Архитектура
Проект следует принципам слоистой архитектуры (Layered Architecture) с использованием паттерна **Фасад** для работы с данными.

### Схема взаимодействия:
1. **User/Admin** посылает сигнал через Telegram API.
2. **Middlewares** выполняют первичную фильтрацию: авто-регистрацию пользователей и стелс-модерацию сообщений в топиках.
3. **Handlers** — Чистая презентация и маршрутизация команд. **Стерильная изоляция**: хендлеры не имеют импортов из `database.db` и взаимодействуют с данными только через сервисы.
4. **Services** — Тонкий слой бизнес-логики и шлюз к данным. Управляют чистотой интерфейса и защищают от спама нажатий.
5. **Database Layer (Facade)** — Обеспечивает атомарный доступ к SQLite через транзакции и WAL-режим. Единственная точка входа для сервисов и клавиатур.

## Ключевые возможности
1. **Стелс-модерация**: топики закрыты по умолчанию — бот беззвучно удаляет сообщения пользователей без прав доступа.
2. **Шаблонное управление доступом**: Доступ управляется исключительно через прямые записи (`direct_topic_access`). Глобальные группы (Templates) служат инструментом для массового копирования прав.
3. **Декларативный Стерильный UI**: декоратор `@UIService.sterile_command` и централизованный навигатор `generic_navigator` позволяют управлять всем интерфейсом бота из единой точки.
4. **Инфраструктура помощи (HelpService)**: Централизованный реестр документации и тултипов. Контент изолирован от кода хендлеров.
5. **Синхронизация имён топиков**: переименование топика в Telegram автоматически отражается в локальной БД.
6. **Интеграция с Google Sheets (The Bridge)**: Асинхронная выгрузка и импорт данных пользователей и групп. Автоматическая фоновая синхронизация при изменениях в БД.
7. **Тройная изоляция навигации**: Весь UI-трафик проходит через `UIService.generic_navigator`, исключая дублирование логики.
8. **Дашборд ролей и FAQ**: общая панель «Роли» для администраторов и модераторов с подробным справочником прав.
9. **Контекстное управление ролями**: назначение прав интегрировано в карточку пользователя.
10. **Единый интерфейс поиска**: при выборе любой сущности автоматически отображается пагинированный список и кнопка 🔎 Поиск.
11. **Нативные оповещения**: рассылка пуш-уведомлений по триггеру `@all`.
12. **Shadow Auto-Registration**: Автоматическое внесение пользователя в БД при первом контакте.
13. **Централизованный ManagementService Layer**: Полная консолидация бизнес-логики (мутации, регистрация, парсинг имен).
14. **Протокол подтверждения (Safety First)**: Деструктивные действия требуют подтверждения.
15. **Типизированная навигация**: Использование констант `PAGINATED_CMDS` гарантирует стабильность интерфейса.
16. **Бронированная БД**: Усиленная схема с Foreign Keys и оптимизированные индексы.
17. **Высокая производительность (Batch-Fetch)**: Пакетная выборка данных исключает проблему N+1 запросов. [PL-HI]
18. **Протокол Экспедиций**: Полный цикл управления мероприятиями клуба (создание, модерация, запись участников).
19. **Защита цепочек ввода**: Системная защита FSM от случайного сброса состояния при пошаговых операциях.
20. **100% Sterile Handlers**: Полное отсутствие прямых связей между хендлерами и БД. Проект защищен от «спагетти-кода» и максимально упрощен для тестирования.
21. **Аудит-слой (Audit & Notification Layer)**: Workflow асинхронного согласования (создание мероприятий, регистрация). Атомарная обработка, идемпотентность, адресные уведомления пользователям о решении админа.
22. **Унифицированный вход (Unified Landing)**: Система «Traffic Controller» роутит пользователей через `/start` на нужную панель (Админ/Модератор/Участник) автоматически. Поддерживает форсированный выбор панели через алиасы `/admin` и `/mod`.
23. **Универсальный протокол индексации**: Каждое правило и паттерн в документации имеют уникальный ID ([CP-x], [PL-x]), что позволяет ассистенту работать с кодом максимально точно, сохраняя чистоту контекста.

## 🛠 Troubleshooting

| Проблема | Причина | Решение |
| :--- | :--- | :--- |
| **RuntimeError: SQLite не поддерживает FK** | Библиотека SQLite в системе устарела или собрана без поддержки связей | Обновите Python/SQLite или смените хостинг. Бот не запустится без гарантии целостности данных |
| **sqlite3.OperationalError: database is locked** | Конкурентная запись в БД | Проверьте, что в `connection.py` включён WAL-режим |
| **Кнопка "Закрыть" не работает** | Потерян `last_menu_id` или стек `last_menu_ids` в FSM | Убедитесь, что используется `state.set_state(None)` |
| **Бот не удаляет сообщения в группе** | Недостаток прав | Бот должен быть администратором с правом «Удаление сообщений» |
| **Топик удалён в Telegram, но висит в БД** | Несовпадение жизненного цикла | Используйте кнопку «Удалить из БД» в меню топиков администратора |
| **Новая БД не работает** | Старый `bot.db` с устаревшей схемой | Удалите файл `database/bot.db` и перезапустите бота — схема пересоздастся автоматически |
| **OperationalError: no such table: topics** | Устаревшие запросы к БД | Таблица была переименована в `topic_names`. Убедитесь, что код обращается к актуальной схеме |
| **TypeError в клавиатурах** | Несоответствие аргументов | После рефакторинга многие клавиатуры (например, `main_admin_kb`) вызываются без аргументов. Проверьте сигнатуры в `UIService` |
| **Бот молчит при вводе данных** | Сброс состояния FSM в промежуточном шаге | Убедитесь, что `UIService.terminate_input` вызывается с `reset_state=False` в середине цепочки ввода |
| **Синхронизация с Sheets не срабатывает** | Вызов синка находился после `return` (мертвый код) | Проблема исправлена: вызов `_trigger_sheets_sync` перенесен внутрь каждой ветки удаления перед `return` |
| **Ошибка NOT NULL в events** | Попытка сохранить пустое название или дату | Введена валидация `message.text`. Проверьте, что пользователь вводит именно текст, а не медиа |
| **Кнопки аудита не исчезают** | Попытка использовать `sterile_show` вместо `delete_msg` для orphan-сообщений | Использовать `UIService.delete_msg(callback.message)` в хендлере решения |

## Настройка и запуск
1. Создайте `.env` и заполните переменные (`BOT_TOKEN`, `ADMIN_ID`, `GROUP_ID`, и опционально параметры Google Sheets).
2. Используйте **Python 3.11**.
3. Создайте виртуальное окружение: `python -m venv venv`.
4. Установите зависимости: `.\venv\Scripts\pip install -r requirements.txt`.
5. Запустите бота через `run_bot.bat` или: `.\venv\Scripts\python main.py`.
